1 · What is this thing?

You tell a computer assistant what you want to find out. Normally it just starts running experiments and reporting numbers. That's fine for a quick look — but for anything you'll stake a decision (or a paper) on, it tends to skip the careful parts. Like a scientist who announces a discovery before checking whether the "discovery" is just measuring something boring.

order-67 is a lab notebook the assistant follows — one that argues back. It makes the assistant:

  • write down the exact, falsifiable claim before running anything — and freeze the plan so it can't be quietly changed later to fit the data,
  • check the data for leaks and hidden confounds before trusting a number,
  • get an independent, skeptical second opinion before calling anything a finding,
  • write up the same verified result two ways: a paper-grade version, and a plain-English brief for whoever decides what to fund next.

It's the research-world sibling of the order-66 software harness. There, the goal was "ship working code." Here, the goal is "establish a claim that survives scrutiny."

You don't run any of this machinery yourself. You type a sentence; the assistant follows the recipe.

2 · The big picture — how a finding gets established

Think of a piece of research as a production line. Each station does one job, then passes the work along. You start by asking a question; the line turns it into a claim you can defend — plus the write-ups that explain it.

Three gates the line will not let you skip. ① Pre-registration: the claim and plan are frozen before any result exists, so you can't quietly move the goalposts. ② Leakage audit: the data is checked for train/test contamination. ③ Confound-audit: every positive result must survive an attempt to explain it away. Plus one rule — the thing that ran the experiment never grades its own result.

3 · Getting set up (one-time)

Step A — install Claude Code

order-67 runs inside Claude Code (the terminal assistant). If you already use it, you're done. If not, install it first and confirm claude opens.

Step B — install the order-67 harness

The harness is a plugin: a folder of recipes Claude Code loads. Point your Claude Code plugin config at the order-67 folder (the same way order-66 is installed). Once it's registered, typing / will offer /triage, /register, and the rest in autocomplete.

Before your first run on a shared repo: the harness's /data-audit scans for secrets committed to the code. If it flags an API key, rotate it — deleting the line isn't enough once it's in git history.

4 · Your first experiment, step by step

You mostly type one sentence at a time and let the assistant drive. A typical run:

Step 1 — ask the question

/triage "Does a wavelet feature of the H-Neuron activations
         tell hallucinated answers from faithful ones?"

The assistant classifies it (is this a quick probe? a full experiment? a replication?) and suggests how careful to be — the rigor tier (§5). It writes nothing until you say go.

Step 2 — pre-register it

/register writes down the falsifiable claim ("…shall reach balanced accuracy ≥ 0.65 on a held-out, question-disjoint split"), names the confounds you already worry about (answer length!), and freezes the plan with a hash. From here on, changing the metric is a deviation the system will flag — not a silent edit.

Step 3 — design & audit the data

/design-experiment writes the plan, starting with the single number you'll look at first. /data-audit then checks the data is split honestly (by question, not by row), nothing leaks from test into train, and no secrets are lying around. If a justification is blank, it stops you.

Step 4 — run it

/plan-runs lays out the runs as a small tree; /execute sends them to your GPU box, watches them, and pulls the results back — capturing the seed, config, and data hash so the run can be reproduced.

Step 5 — try to break your own result

If a result looks positive, /confound-audit runs automatically: it shuffles the labels, checks whether a boring variable (like answer length) explains the effect, and re-runs with a different statistic. /analyze then computes honest statistics — effect sizes and confidence intervals, not lonely p-values.

Step 6 — get the independent verdict

A separate, skeptical reviewer (the results-verifier) reads only the logged numbers and returns one of: PROMOTE, PROMOTE-WITH-CAVEAT, RERUN, REJECT, or BLOCKED. The assistant that ran the experiment never gets to grade it.

Step 7 — write it up (two ways)

On a PROMOTE, /paper drafts a paper-grade write-up (every claim carries its statistics) and /brief drafts a one-page, plain-English brief for management — a bottom line, a letter grade, honest caveats, and a "here's what a skeptic will say, and here's our answer" handbook.

You can stop anywhere. A clean negative result is a finding too — order-67 records nulls instead of burying them.

5 · Four rigor tiers — picking how careful to be

Not every question deserves a six-month confirmatory study, and not every result should be trusted because it "looked converged." order-67 has one dial — the rigor tier — and it decides how many checks apply. Each tier adds to the one below; you never get production-grade ceremony forced onto a quick look.

The safety rule: if the tier (or a seed, or the frozen plan) is missing, the harness assumes the strictest tier and says so out loud. A result with no metadata is treated as suspicious, never waved through.

6 · The skills, explained simply

Each /word is one station. You rarely call them all by hand — the assistant chains them — but here's what each does.

/triage
Sorts your question into the right path and suggests a rigor tier. The "where does this go?" station.
/register
Writes the falsifiable hypothesis and freezes the plan with a hash. The pre-registration gate.
/design-experiment
Writes the method — starting with the one number you'll eyeball first — plus the data & compute plan and the threats to validity.
/data-audit
Checks for leakage and confounds and scans for committed secrets. Stops you on a blank justification.
/plan-runs
Lays out the runs as a small tree, with a budget cap and the first task being the quick "is there a pulse?" look.
/conflicts
Checks this experiment against project-wide rules (e.g. "never import the test set") before spending GPU.
/execute
Runs the experiments on your remote GPU box, watches them, fetches results, and records how to reproduce each run.
/confound-audit
Tries to explain away every positive result (permutation null, nuisance screen, statistic swap). The "try to break it" station.
/analyze
Computes honest statistics (effect size + confidence interval + multiple-test correction) and commissions clean figures.
/verify-claim
Asks the independent reviewer for a verdict — on demand, when you want to re-check.
/paper
Drafts the paper-grade write-up; every claim carries its statistics. Runs a NeurIPS-style checklist on itself.
/brief
Drafts the plain-English brief for management: bottom line, letter grade, caveats, and a pushback handbook.
/drift
Tidies up: consolidates notes, flags figures whose finding was retracted, chases overdue "rigor debt."
/note
Jots a timestamped thought into today's notes without interrupting the run.

7 · Why an independent skeptic matters

The single most valuable thing order-67 does is refuse to let the assistant grade its own homework. The thing that ran the experiment is motivated to see a result; a separate reviewer, reading only the logged numbers, is not.

Here's a real one, baked into the repo as a worked example (examples/spectral-length-bias/):

The 71% that wasn't. A wavelet feature of neuron activations seemed to detect hallucinated answers — 71% balanced accuracy on a held-out, honestly-split set. The data audit was clean: no leakage. It looked like a finding.

Then the confound-audit went to work. Shuffling the labels confirmed the feature was predicting something (good). But the nuisance screen asked a boring question: are hallucinated answers just longer? They were — 142 tokens versus 38 — and the feature was ~0.8 correlated with length. Control for length, and the 71% collapsed to 53% — chance. Swap in a length-blind statistic and the effect vanished entirely.

Verdict: REJECT. The reviewer marked the claim "not supported — the detector detects answer length, not hallucination," and the experiment's status flipped to retracted. The harness reached, by mechanism and in minutes, the same conclusion the human team had reached the hard way.

Note the two gates did different jobs: /data-audit said PASS (the split really was clean), while /confound-audit said the result doesn't survive. Leakage and confounding are different diseases — order-67 checks for both.

8 · What lives where on your computer

Inside a research project, the harness grows a tidy set of folders. You can read any of these in a text editor.

your-project/
  experiments/<exp>/        one folder per experiment
    experiment.json          the experiment's "state" (tier, status, hashes)
    hypothesis.md            the falsifiable claim(s)
    prereg.lock              the frozen plan (hashed — tamper-evident)
    preregistration.md       the readable pre-registration
    design.md  runs.md       the method and the run tree
    analysis.md  audit.md     the stats, and the confound-audit
    verdict.md               the independent reviewer's call
    runs/<RUN>/              each run's seed, config, metrics, logs
  results/<exp>/             figures + machine-readable stats
  trace/<exp>.md             append-only log of every claim → file:value
  FINDINGS.md                short index of what's been established
  RIGOR_DEBT.md              shortcuts you owe, with due dates
  sessions/                  daily notes & metrics

Two ideas worth knowing: the trace is the paper trail — every figure and number can be tracked back to the run that produced it. And the findings index is deliberately short, so the project's memory doesn't rot into a swamp.

9 · When things look weird

"The skills don't show up when I type /"

The plugin isn't registered. Re-check Step B; confirm Claude Code lists order-67 among its plugins, then restart the session.

"It refuses my hypothesis"

/register rejects claims that can't be proven wrong ("the samples look good"). Restate it as a number on held-out data: "…shall reach metric ≥ threshold on the held-out split." If you can't, that's a sign the question isn't ready to test yet.

"The verifier said REJECT (or my result got retracted)"

That's the system working, not breaking. Read verdict.md — it cites exactly which check failed and where. Usually it's a confound (like length) or a missing control. Fix the design, re-register if the plan changed, and run again.

"It says BLOCKED, not REJECT"

Different thing. REJECT = the number exists but doesn't support the claim. BLOCKED = the number was never actually logged, so nothing can be judged. Log the metric and re-run.

"A run won't promote at pilot — it wants a confound screen"

By design. Pilot is the gate to expensive confirmatory work, so a positive there must at least pass the quick length/permutation check before you're allowed to spend big. It's protecting your GPU budget from a mirage.

"Files I didn't ask for keep appearing"

That's the trace hook recording provenance as you write. It's append-only and harmless; /drift keeps it tidy.

10 · Under the hood — repo layout

For people extending the harness itself. The repo is a single-plugin marketplace: the marketplace manifest sits at the root, and the installable plugin lives in plugin/ (the same shape as order-66, so claude plugin install …@order-67 works).

order-67/                          the marketplace repo
  .claude-plugin/marketplace.json  exposes the plugin (source: ./plugin)
  README.md  ROADMAP.md  docs/     repo-level docs (this guide lives here)
  examples/spectral-length-bias/   the worked acceptance experiment
  plugin/                          the installable plugin (copied to cache on install)
    .claude-plugin/plugin.json     manifest + hook wiring
    CONVENTIONS.md                 the contract everything obeys
    skills/<name>/SKILL.md         one folder per skill (+ templates/)
    agents/<name>/AGENT.md         the 6 subagents
    hooks/                          update-trace.sh, block-lint.sh, …
    templates/                      rulebook, schemas, host-config example

Install it at user scope with:

claude plugin marketplace add /path/to/order-67
claude plugin install order-67-research-harness@order-67   # --scope user (default)

11 · The ## Block contract & the linter

Skills talk to each other by emitting a named markdown block (e.g. ## Verdict) whose first lines are mechanical — - Field: value — so the next skill can read them by grep. The required fields per block live in hooks/block-schemas.tsv, and hooks/block-lint.sh checks them: not just that a field is present, but that load-bearing values are valid (a verdict is one of the five allowed strings; the leakage justifications aren't blank). It runs as an advisory hook on every write, and as a hard gate inside /execute — so a typo can't waste a GPU run.

12 · Anatomy of a skill

Every SKILL.md has frontmatter (name, description, whenToUse) then two parts: Static (Summary, Preconditions, Parameters, the exact output block, Hard constraints, Common mistakes) and Dynamic (template references and !{bash} snippets the assistant runs at the moment the skill fires). The constraints and common-mistakes sections are where the rigor lives.

13 · Anatomy of a subagent

The six agents (results-verifier, data-steward, reproducibility-checker, peer-reviewer, figure-smith, run-reader) are isolated reviewers with their own tools and model. Crucially, the graders are write-disabled: the verifier can read and reason but cannot edit the experiment it's judging. That's how "never self-grade" is enforced structurally, not just asked for politely.

14 · Hooks — events & the trace

Two events are wired in plugin.json. On every file write under experiments/ or results/, update-trace.sh appends a provenance line and block-lint.sh checks any block in the file. On session stop, a logger records the event (automatic consolidation is deliberately left manual for now — run /drift yourself, so missing tidy-up is visible rather than silently skipped). The hooks use no absolute paths and don't depend on jq, so the plugin is portable.
